Étape 2 / 8 · Préparation

Installer Git
sur votre ordinateur

Git est l’outil qui synchronise vos fichiers locaux avec GitHub. Il est indispensable pour envoyer votre code sur Vercel.

❓ Git est-il déjà installé ?
Ouvrez PowerShell et tapez cette commande :
git --version
✅ Si vous voyez
git version 2.x.x
Git est installé → passez directement à la configuration ci-dessous.
❌ Si vous voyez une erreur
"command not found"
Git n’est pas installé → suivez les instructions ci-dessous.
1 Télécharger et installer Git
1

Ouvrir le site officiel Git

git-scm.com/download/win → Le téléchargement démarre automatiquement (fichier Git-x.x.x-64-bit.exe)

2

Lancer l’installateur

Double-clic sur le fichier téléchargé → Cliquez "Next" à chaque étape (options par défaut correctes) → "Install" → "Finish"

3

Vérifier l’installation

Fermez et rouvrez PowerShell → git --version → vous devez voir git version 2.x.x

2 Configurer Git (une seule fois)
Git a besoin de connaître votre identité pour signer chaque modification. Utilisez la même adresse email que votre compte GitHub.
# Remplacez par vos vraies informations git config --global user.name "Votre Nom" git config --global user.email "[email protected]" git config --global init.defaultBranch main # Vérifier : git config --global --list
3 Créer un token d’accès GitHub (PAT)
GitHub n’accepte plus les mots de passe pour les pushes depuis PowerShell. Le Personal Access Token remplace votre mot de passe pour git push.
1

Ouvrir les paramètres GitHub

github.com/settings/tokens/new (connectez-vous si besoin)

2

Configurer le token

• Note : nexus-market-token
• Expiration : 90 days
• Scopes : cochez ✅ repo (tout cocher dans repo)
• Cliquez "Generate token"

3

Copier et sauvegarder le token

GitHub l’affiche une seule fois (commence par ghp_...). Copiez-le dans un endroit sûr — vous l’utiliserez comme mot de passe lors du premier push.

🔐
Ne partagez jamais ce token ! Il donne accès complet à vos dépôts. Si perdu, créez-en un nouveau depuis GitHub Settings → Developer settings → Personal access tokens.

Étape 4 / 8 · Préparation

Envoyer vos fichiers
sur GitHub

Initialisez Git dans votre dossier nexus-market et envoyez tous vos fichiers sur GitHub pour la première fois.

1 Créer le .gitignore en premier
Avant tout, créez ce fichier pour empêcher vos clés secrètes d’aller sur GitHub. Sauvegardez-le en tant que .gitignore dans votre dossier nexus-market.
.gitignore — C:\Users\pheni\Downloads\nexus-market\.gitignore
# ── Secrets — ne JAMAIS envoyer sur GitHub ─── .env .env.backup .env.local .env.*.local # ── Dépendances npm ───────────────────── node_modules/ npm-debug.log* # ── Logs ───────────────────────────── *.log logs/ # ── Système Windows ───────────────────── Thumbs.db desktop.ini # ── Fichier reset démo (dangereux en prod) ── nexus_reset.html .env.backup
🔐
Critique : si .env n’est pas dans .gitignore, vos clés Supabase, Stripe et SMTP seront visibles publiquement sur GitHub — une faille de sécurité critique.
💡
Pour créer .gitignore : Bloc-notes → Fichier → Enregistrer sous → Type : "Tous les fichiers" → Nom : .gitignore (avec le point au début, pas d’extension .txt)
2 Premier push — commandes PowerShell
Copiez et exécutez ces commandes une par une dans PowerShell :
1

Aller dans le dossier

cd "C:\Users\pheni\Downloads\nexus-market"
2

Initialiser Git

git init # Résultat attendu : "Initialized empty Git repository..."
3

Vérifier que .env n’apparaît pas

git status # .env ne doit PAS apparaître dans la liste # Si .env apparaîet → vérifiez votre .gitignore
4

Ajouter et commiter tous les fichiers

git add . git commit -m "Initial commit — NEXUS Market v3.1"
5

Lier au dépôt GitHub (remplacez USERNAME)

git remote add origin https://github.com/USERNAME/nexus-market.git git remote -v
▲

Premier push vers GitHub

git push -u origin main
💬 GitHub demande votre username GitHub
🔐 GitHub demande votre "password" → entrez votre token PAT (ghp_...), pas votre vrai mot de passe
Le curseur ne bouge pas quand vous tapez le token — c’est normal (caractères masqués)
✅
Si tout va bien : Branch 'main' set up to track remote branch 'main' from 'origin'. Vérifiez sur GitHub.com que vos fichiers sont bien visibles !

🚨 Problèmes fréquents
❌

"Support for password authentication was removed"

Vous avez tapé votre mot de passe GitHub. Relancez git push et entrez votre token PAT (ghp_...) au lieu du mot de passe.

❌

"error: remote origin already exists"

Supprimez l’ancien remote : git remote remove origin puis recréez-le avec git remote add origin https://...

❌

"src refspec main does not match any"

Vous n’avez pas encore fait de commit. Faites d’abord git add . puis git commit -m "init"

❌

"Please tell me who you are"

Exécutez les commandes git config --global user.email et git config --global user.name de l'étape 1.

Étape 5 / 8 · Préparation

Préparer les fichiers
pour Vercel

Vercel a besoin d'un fichier de configuration minimal et d'un ajustement de votre frontend pour pointer vers l'API en production.

1 Créer vercel.json (configuration Vercel)
Créez un fichier vercel.json à la racine de votre dossier nexus-market avec ce contenu :
vercel.json
// Fichier : C:\Users\pheni\Downloads\nexus-market\vercel.json { "version": 2, "builds": [ { "src": "*.html", "use": "@vercel/static" }, { "src": "server.js", "use": "@vercel/node" } ], "routes": [ { "src": "/api/(.*)", "dest": "/server.js" }, { "src": "/webhooks/(.*)", "dest": "/server.js" }, { "src": "/(.*\\.html)", "dest": "/$1" }, { "src": "/(.*)", "dest": "/nexus_market.html" } ] }
ℹ️
Ce fichier dit à Vercel : les fichiers .html sont du contenu statique, et server.js est l'API backend.
2 Créer un fichier .gitignore
Créez .gitignore pour ne pas envoyer les fichiers sensibles sur GitHub :
.gitignore
# Fichier : C:\Users\pheni\Downloads\nexus-market\.gitignore .env .env.backup node_modules/ .env.local *.log
🔒
Critique : le fichier .env contient vos clés secrètes. Sans .gitignore, il serait visible publiquement sur GitHub.
3 Mettre à jour NEXUS_CONFIG dans nexus_market.html
Cherchez la section NEXUS_CONFIG dans nexus_market.html (~ligne 975) et mettez à jour l'URL de l'API :
nexus_market.html — NEXUS_CONFIG
// Remplacer localhost par votre URL Vercel const NEXUS_CONFIG = { apiUrl: "https://nexus-market.vercel.app", stripe: { publishableKey: "pk_test_VOTRE_CLE_STRIPE" } };
Format : https://votre-projet.vercel.app
4 Pousser les modifications sur GitHub
Après avoir créé ces fichiers, envoyez-les sur GitHub :
git add vercel.json .gitignore git commit -m "Add Vercel config" git push
💡
Avec GitHub Desktop : cliquez le bouton "Commit to main" puis "Push origin"
Étape 6 / 8 · Configuration

Variables d'environnement
sur Vercel

Vos clés secrètes ne doivent jamais aller sur GitHub. Vous les entrez directement dans le tableau de bord Vercel.

🗺️ Où trouver chaque clé — guide complet
Cliquez sur chaque service pour voir comment récupérer vos clés :
🟢
Supabase — SUPABASE_URL + SUPABASE_SERVICE_KEY
Obligatoire • Base de données de votre application
1

Ouvrir votre projet Supabase

supabase.com/dashboard → cliquez sur votre projet

2

Aller dans Settings → API

Dans le menu gauche → Settings (icône engrenage) → API

3

Copier les valeurs

• SUPABASE_URL → champ "Project URL" (commence par https://xxxx.supabase.co)
• SUPABASE_SERVICE_KEY → section "Service role" (commence par eyJ...) — PAS l'anon key !

⚠️
Utilisez la Service role key (pas la anon/public key). La service role donne un accès complet à la BDD — ne la partagez jamais publiquement.
🔑
JWT_SECRET — Générer une clé sécurisée
Obligatoire • Clé de chiffrement pour les tokens d'authentification
Le JWT Secret est une longue chaîne aléatoire que vous créez vous-même. Il doit faire au moins 32 caractères. Utilisez une de ces méthodes :
💻 PowerShell
# Génère 64 chars aléatoires -join ((65..90)+(97..122)+(48..57) | Get-Random -Count 64 | %{[char]$_})
🌐 En ligne (plus simple)
→ generate-secret.vercel.app/64
→ Copiez la valeur générée
→ C'est votre JWT_SECRET
⚠️
Sauvegardez ce secret dans un endroit sûr ! Si vous le perdez, tous les utilisateurs devront se reconnecter.
💳
Stripe — STRIPE_SECRET_KEY (optionnel)
Optionnel • Nécessaire seulement si vous acceptez des paiements
1

Ouvrir le Dashboard Stripe

dashboard.stripe.com/test/apikeys

2

Copier la clé secrète de test

Secret key → cliquez "Reveal test key" → copiez la valeur (commence par sk_test_...)

💡
Commencez avec la clé sk_test_ (test). En production, remplacez par sk_live_ uniquement quand tout fonctionne.
📧
Gmail SMTP — SMTP_USER + SMTP_PASS (optionnel)
Optionnel • Pour l'envoi d'emails de confirmation
1

Activer la validation en 2 étapes Gmail

myaccount.google.com/security → "Validation en 2 étapes" → activer si pas déjà fait

2

Créer un "Mot de passe d'application"

myaccount.google.com/apppasswords → Sélectionner "Autre" → Nom : "nexus-market" → "Générer"

3

Copier le mot de passe généré

Gmail affiche un code en 4 blocs de 4 lettres (xxxx xxxx xxxx xxxx) → c'est votre SMTP_PASS. Votre vrai mot de passe Gmail ne fonctionne pas ici.

📝 Entrez vos valeurs pour générer la liste
⚙️ Variables à ajouter dans Vercel
Vercel → votre projet → Settings → Environment Variables → ajoutez chaque ligne :
⚠️
Dans Vercel, pour chaque variable : copiez le nom dans "Key", copiez la valeur dans "Value", cliquez Save. Répétez pour chaque ligne.

🚨 Erreurs fréquentes lors du déploiement
❌

"Build failed" — Échec de compilation

Cause : Vercel ne trouve pas package.json ou il y a une erreur de syntaxe dans votre code.
Solution : Vérifiez que package.json est à la racine de votre dépôt GitHub (pas dans un sous-dossier). Cliquez sur les logs rouges dans Vercel pour voir l'erreur exacte.

❌

"Function Invocation Failed" — l'API ne répond pas

Cause : Les variables d'environnement sont manquantes ou incorrectes (SUPABASE_URL, JWT_SECRET...).
Solution : Allez dans votre projet Vercel → Settings → Environment Variables → vérifiez que toutes les variables obligatoires sont présentes → puis Deployments → Redeploy.

❌

"404 Not Found" sur /api/health

Cause : Le fichier vercel.json est absent ou mal configuré.
Solution : Vérifiez que vercel.json est bien visible sur GitHub à la racine de votre dépôt. Si absent, revenez à l'étape 5 pour le créer et le pousser.

❌

"Cannot find module" dans les logs

Cause : Un package npm utilisé dans server.js n'est pas listé dans package.json.
Solution : Ajoutez le package manquant dans package.json → pushez → Vercel redéploie.

❌

Site visible mais l'API répond "Unauthorized" ou erreur 500

Cause : JWT_SECRET incorrect ou SUPABASE_SERVICE_KEY mal copié (souvent des espaces en trop).
Solution : Dans Vercel → Settings → Environment Variables → supprimez et re-collez chaque valeur en faisant attention aux espaces. Redéployez.

❌

Le déploiement réussit mais le site affiche une page blanche

Cause : NEXUS_CONFIG dans nexus_market.html pointe encore vers localhost.
Solution : Mettez à jour apiUrl avec votre URL Vercel (étape 5 ci-dessus) puis pushez.

# Modifier vos fichiers localement # Puis dans PowerShell : git add . git commit -m "Description de votre modification" git push # → Vercel redéploie automatiquement en ~20s
🔧 Prochaines étapes suggérées
→

Configurer un domaine personnalisé

Vercel → Settings → Domains → ajouter nexus.sn

→

Activer Stripe en production

Remplacer sk_test_ par sk_live_ dans les variables Vercel

→

Configurer les webhooks Stripe en prod

Stripe dashboard → Webhooks → ajouter https://nexus-market.vercel.app/webhooks/stripe

→

Supprimer nexus_reset.html du dépôt

Ce fichier réinitialise toutes les données — il ne doit pas être accessible en ligne.
