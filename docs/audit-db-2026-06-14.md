## ✅ Documentation corrigée (CLAUDE.md)

L'audit a montré que `CLAUDE.md` était **factuellement obsolète** sur plusieurs points :

- `orders.status` inclut aussi **`pending`** (pas seulement `pending_payment`…).
- `orders.payment_method` inclut **`cod`** (paiement à la livraison), pas seulement `{card, mobile}`.
- `notifications.type` autorise **`new_vendor`** (8 valeurs, pas 7) — la contrainte live le permet,
  donc les 2 lignes `new_vendor` présentes ne sont **pas** des erreurs.
- Les colonnes dites « inexistantes » (`amount_eur`, `paid_at`, `paytech_token`, `failure_reason`)
  **existent bel et bien** en prod (dette legacy).

## ⚠️ À trancher (NON corrigé — décision métier / revue code requise)

Ces points sont des **incohérences réelles** mais leur correction est destructive ou exige un
arbitrage : je ne les ai pas appliqués pour ne pas risquer de casser la prod.

1. **Colonnes montant redondantes sur `orders`** : `total`, `order_total`, `amount_eur`,
   `amount_fcfa`, `subtotal` coexistent. **Incohérence critique** : les valeurs de `total`
   observées sont en **EUR** (ex. `36.13`, `13.14`) alors que la doc et le reste du code
   supposent du **FCFA**. → Décider la colonne/devise canonique, convertir, puis supprimer
   les colonnes mortes. Ne pas toucher aux montants sans validation (données financières).

2. **Colonnes legacy dupliquées sur `orders`** : `user_id` (doublon de `buyer_id`),
   `id_old` (doublon de `id`), `canceled_at` (doublon de `cancelled_at`). → Après revue du
   code (certains chemins écrivent peut-être encore `user_id`), les déposer.

3. **Deux jeux de migrations divergents** (`database/migrations/` vs `sql/`) — cf. CLAUDE.md §4.
   Le live tranche en faveur de `orders.id = uuid` et `loyalty_points = table de solde`
   (`points/total_earned/total_redeemed`). Recommandé : aligner les fichiers sur le live.

## Manquements (lacunes) — analyse complémentaire

### ✅ Comblé (sûr)
- **46 colonnes FK sans index** → indexées. Migration `2026_06_14_fk_indexes.sql` (+ `sql/`), appliquée.
  Postgres n'indexe pas les FK automatiquement ; sans index, jointures et `ON DELETE CASCADE/SET NULL`
  scannent toute la table enfant. (ex. `deliveries.order_id`, `courier_earnings.*`, `stripe_sessions.*`,
  `loyalty_history.order_id`, `reviews.vendor_id`, `pro_reviews.user_id`…). 0 restante après coup.

### ✅ Aucun manquement (vérifié, rassurant)
- **RPC** : les 37 RPC appelées par le code existent toutes en base (dont `create_order_atomic`,
  `api_key_validate`, `nearby_pros`…) — contrairement à d'anciennes notes. Aucune RPC manquante.
- **RLS** : activée sur **100 %** des tables.
- **Clés primaires** : présentes partout (sauf la table de backup `_orders_backup_20260521`).

### ⚠️ À trancher (risqué)
- **Surcharges de fonctions ambiguës** (dérive text↔uuid des 2 jeux de migrations) : `save_invoice`,
  `cancel_order_release_stock`, `add_loyalty_points` (×4), `search_products` ont des variantes de
  **même arité** différant par `p_order_id text` vs `uuid` (ou `double precision` vs `numeric`).
  PostgREST peut renvoyer « could not choose best candidate function ». Comme `orders.id = uuid`,
  les variantes `text` sont probablement mortes → les supprimer **après** vérification des sites d'appel.
- **`_orders_backup_20260521`** : table de backup obsolète (16 lignes, sans PK) qui encombre le schéma
  → à supprimer si la sauvegarde n'est plus utile.
